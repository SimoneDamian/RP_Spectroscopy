----- New parameters page instructions -----
I modified the structure of the writeable parameters in the .yaml file that contains the parameters of the Red Pitaya (you can find an example of the new version in @RedPitaya_K_parameters copy.yaml that will be renamed RedPitaya_K_parameters.yaml, also the default version of the file will be substituted with the new form factor) and I want to update the load_parameters method inside the @service_manager.py in order to read it correctly (I simply added more entries for each parameter, so it should not be modified). The entries are: hardware_name, gui_name, description that are strings, type that can be int, float or bool, initial_value, min, max, that are numbers of the type aforementioned or can also be null, scaling that are, in general, float and units that can be a string or can be null.
Then the Service Manager sends the dictionary to the General Manager that handles the GUI and sends the same dictionary to the Laser Manager (already implemented).
I want to change the visualization of the parameters table in the GUI. The page is the ParametersPage class in the @laser_controller_page.py. In this new version I want to have these columns: ["Name", "Value", "Units", "Min", "Max"] that are linked to the entries of the dictionary elements [gui_name, value, units, min, max]. As it is now, only the "Value" column is editable by the user.
In addition, I want the user to be able to see the "description" of a parameter when the arrow stays on it. All the rest should be ok as it is now, considering the fact that when the app closes the Service Manager writes only the value in the YAML file.

----- New advanced settings page instructions -----
I rewritten the advanced settings part of the .yaml file linked to the specific board one connects to (have a look at @RedPitaya_K_parameters copy.yaml). This file is red in the load_default_advanced_settings method contained in the service_manager.py file. I want you to check if with these new modifications the file can still be red.
I want to modify the visualization of these settings in the @advanced_settings_page.py. I still want a scrollable list of QGroupBox widgets, one per ui_label group found in the advanced_settings section of the board YAML but now with a specific format for each group. For the unlock_detection group: the title is the ui_label one. Then I want a drop-down menu for each one of the other elements (events, fast_control, slow_control, monitor_signal), each one named by its gui_name. Inside this drop-down menu I want another drop-down menu for each entry with the associated gui_name. Inside this I want a row with the other informations like a checkbox for the enabled status and, where it is present, a textbox for the threshold or for the num_points. As before, the only thing that the user is allowed to modify is the enabled status, the num_points and the threshold entries.
For the gui_visualization group: The title of the group is the ui_label. For each other element (history_length, fast_control_signal_derivative, slow_control_signal_derivative) I want a drop-down menu labeled with the associated gui_name that contains a checkbox for the enabled entries and a textbox for the fast and slow history_lengths (followed by the units). Only the fast, slow and enabled entries can be modified by the user.
For the autolock_settings group: The title of the group is the ui_label. In this case I want only two rows, one for mode and one for determine_offset, each one with its gui_name. Then, in the mode row I want a dropdown menu from which the user can choose between the available options. For the determine_offset I want only a checkbox for the enabled value.
For the autocenter_settings group: The title of the group is the ui_label. For each element (jitter_threshold, threshold_count, offset_big_jump, offset_small_jump, offset_try_list, correlation_minimum, length_match_minimum, proportion_free_space_left) I want a drop-down menu labeled with the associated gui_name that contains a textbox where the user can modify the value of the parameter.
In addition, I want the user to be able to see the "description" of a setting when the arrow stays on it.

----- Manual lock instructions -----
Main Menu -> Manual lock button -> left page as drawings, right page with the plots as in the SWEEP state but with the addition of a red vertical line for the x0 and x1 with a red area between them.
When start lock button pressed: start locking procedure showing on the right the "Start locking procedure..." snd on the right returning to the main menu, the FSM of the LM is set to IDLE.
When the check_for_lock_status gives True, the FSM is set on the LOCKED mode that will send to the GUI the package with the infos to show  on the right in the locking monitor.
When the FSM is in the LOCKED state the lock_check method is triggered at each cycle. It has to: check the lock status with the selected methods, send the histories to the GUI. If the lock is lost, the FST should pass in the SWEEP mode.
The history is managed by the LaserManager, It receives the history from the Interface and it fills the actual history that is shown to the user with the appropriate length.
